🔥 Hot Code Reloading System - Build Roadmap

1

Foundation: Understanding the Pieces

Module System Basics

Explore how Python's import system works:

  • What happens when you import module?
  • Where do modules live? (sys.modules)
  • What's the difference between importing a module vs importing from a module?

The Reference Problem

Create a simple test case:

  • Module A defines a function
  • Module B imports and stores reference to it
  • Reload Module A
  • What happens to B's reference?

importlib.reload()

Experiment with Python's built-in reloading:

  • What does it do?
  • What doesn't it handle?
  • When does it fail?

Experiments to Try

  • Write two modules where one imports from the other
  • Store a reference to a function in a variable
  • Reload the source module and see if the reference updates
  • Try with functions, classes, and class instances

Milestone

You should understand WHY naive reloading breaks and what the core challenge is.

AST Analysis: Finding All Callables

Parse and Inspect

Learn to traverse Python's AST:

  • Parse a simple module with ast.parse()
  • Print the tree structure
  • Identify different node types

The Visitor Pattern

Build your first NodeVisitor:

  • Visit all function definitions
  • Visit all class definitions
  • Print their names and line numbers

Context Tracking

The key challenge:

  • How to distinguish functions from methods?
  • Use instance variables to track state
  • Implement save/restore for nested classes

Challenge: Nested Classes

Can your visitor correctly extract methods from nested classes without mixing them up?

Test with a class that has a nested class inside it, where both have methods.

Think About

When you enter a class, you need to remember you're "inside" it. When you exit, you need to remember what you were doing before. Sound familiar? Think about how function call stacks work!

Questions to Answer

  • What information do you need to store about each callable?
  • How will you uniquely identify a method vs a function with the same name?
  • What about lambdas? How do they appear in the AST?
  • Can you find where a function is defined (line number)?

Milestone

You can scan any Python module and produce a complete list of all functions, classes, and methods with their locations.

3

Dependency Tracking: Who Imports What?

Import Hooks

Intercept the import process:

  • Research Python's import hooks
  • Look at __builtins__.__import__
  • Figure out how to wrap it

Dependency Graph

Build a graph data structure:

  • What nodes represent
  • What edges represent
  • How to traverse it

Transitive Dependencies

The chain reaction:

  • If A imports B, and B imports C...
  • When C changes, what needs reloading?
  • How to find all affected modules?

Build This

Create a system that logs every import. When Module A imports Module B, record it. Build a directed graph where edges point from "importer" to "imported".

Challenge: Circular Imports

What happens when Module A imports B, and B imports A? How will your graph handle this? Can you detect cycles?

Questions to Answer

  • Should you track ALL imports or just your project's code?
  • How do you distinguish between import x and from x import y?
  • What about dynamic imports (importlib.import_module())?
  • Do you need to track the ORDER of imports?

Milestone

You can visualize your project's import graph and find all modules that would be affected by changing any given module.

4

Runtime Call Graph: Who Calls What?

Execution Tracking

Track function calls at runtime:

  • Decorator pattern for wrapping
  • Maintaining a call stack
  • Recording caller/callee relationships

Graph Node Design

What should each node store?

  • The callable itself
  • Who called it
  • What it called
  • Cached results?

Dynamic vs Static

Combine two views:

  • AST gives you static structure
  • Runtime gives you actual behavior
  • How to merge them?

Key Insight

The AST tells you what CAN be called. The call graph tells you what WAS called. Both are important! You need the first to know what to reload, and the second to know what to invalidate.

Challenge: Recursive Functions

How do you handle a function that calls itself? Should you create multiple nodes or track it differently?

Design Decisions

  • Do you wrap ALL functions or let users opt-in?
  • What's the overhead of tracking? Is it acceptable?
  • Should you track arguments to function calls?
  • How do you handle callbacks and higher-order functions?

Milestone

You can wrap functions and build a runtime graph showing which functions called which during execution.

5

The Registry: Central Source of Truth

Lookup Table Design

How to store and retrieve callables:

  • What's the key? (module name + qualified name?)
  • What's the value? (the callable object)
  • How to handle updates?

Indirection Layer

The solution to the reference problem:

  • Don't store direct references
  • Store lookups instead
  • Always fetch latest at call time

Proxies or Smart Pointers?

Different approaches:

  • Proxy objects that forward calls
  • Registry lookups at call time
  • What are the tradeoffs?

Prototype This

Build a simple registry that can store functions by name and retrieve them. Then build a proxy that always calls the latest version from the registry. Test that updating the registry makes the proxy call new code.

Think About Transparency

Can you make proxies that look exactly like the real functions? Consider __call__, __getattr__, and functools.wraps.

Milestone

You have a registry that can store and update callables, and a way to always get the latest version without broken references.

6

Integration: Putting It All Together

The Reload Pipeline

When a file changes:

  1. Detect which module changed
  2. Use dependency graph to find affected modules
  3. Reload modules in correct order
  4. Re-scan with AST extractor
  5. Update registry with new callables
  6. Invalidate call graph caches

State Management

The tricky part - what persists?

  • Which results should be cached?
  • Which should be recomputed?
  • How does user control this?

User Interface

How do users interact with your system?

  • Decorators for marking functions?
  • Context managers for sessions?
  • Explicit reload calls?

The Big Challenge: Correctness

How do you ensure reloading happens in the right order? If Module A depends on B and C, and C depends on B, what's the correct reload sequence? This is a topological sort problem!

Architecture Questions

  • Should the system auto-reload on file changes or wait for manual trigger?
  • How do you handle errors during reload? Roll back?
  • What's the API for freezing/unfreezing cached results?
  • How do you integrate with existing code minimally?

Milestone

You can reload a changed module and all its dependencies correctly, with the call graph and registry updating automatically.

7

Polish: Edge Cases and Features

Error Handling

What could go wrong?

  • Syntax errors in reloaded code
  • Import errors
  • Runtime errors during init
  • How to recover gracefully?

Advanced Cases

Can you handle:

  • Class instances (updating methods on live objects)
  • Closures and nested functions
  • Decorators and wrapped functions
  • Async functions and coroutines

Developer Experience

Make it pleasant to use:

  • Clear error messages
  • Visual feedback (what was reloaded?)
  • Debugging tools (view graphs)
  • Performance monitoring

Testing Ideas

  • Create a deeply nested module structure and reload leaves
  • Test with circular dependencies
  • Try reloading during active execution
  • Benchmark the overhead of tracking

Final Milestone

You have a robust hot reloading system that handles edge cases gracefully and provides a good developer experience.

Learning by Doing

Remember: The goal isn't to build the perfect system on the first try. It's to understand the problem deeply.

Iterate: Build the simplest version first, then add complexity. Start with just functions, then add classes, then methods, then lambdas.

Experiment: Break things! Try to make your system fail. That's how you discover what needs to be handled.

Question everything: Why this data structure? Why this approach? What are the alternatives? What are the tradeoffs?

The best way to learn is to struggle with the problem yourself. 🚀
